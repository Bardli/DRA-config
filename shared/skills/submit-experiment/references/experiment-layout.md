# Experiment Record Layout & `metadata.yaml` Schema (canonical)

The single definition of the experiment-record structure. `submit-experiment` (creates records)
and `harvest` (resolves status, regenerates the index) both follow this — keep them in sync with
this file, not with each other.

**Two artifacts per run:** `metadata.yaml` holds the *metadata* (structured, machine truth —
status / provenance / metrics); a narrative `.md` holds the *story* (Goal / Setup / Results /
Observations). **`md` = narrative, `yaml` = metadata** — status lives in the yaml and is never
parsed from the prose.

**What you record is per-project.** This is a *complete reference example* distilled from a real
training project; keep the **Core** fields always, add **Reproducibility** fields whenever a run
must be re-creatable, and adopt the **Scientific / lineage** fields as your work warrants. Trim to
taste — but never drop the SSOT + provenance core.

## Directory layout (durable vs scratch)

For real ML projects the whole `experiment/` tree usually lives in `$SCRATCH` (purgeable). Split
**durable record** (small, back up to `$PROJECT` or a records repo) from **large/regenerable**
(stays in scratch; referenced by `metadata.yaml`, never committed).

```
experiment/<run_code>/
  # ---- durable record (small; back up / version-control-friendly) ----
  metadata.yaml          # SSOT (status, provenance, objective, result)
  config.yaml            # input config snapshot (as submitted)
  config_resolved.yaml   # fully-resolved config (Hydra/overrides applied) — the ACTUAL config used
  slurm_*.sh             # submit script snapshot(s): e.g. slurm_smoke.sh, slurm_train_eval.sh, run_local.sh
  git.diff               # saved working-tree diff when the run was dirty (see git.diff_path)
  eval_<variant>/        # evaluation outputs, e.g. dice_*.csv (small, keep)
  # ---- large / regenerable (live in $SCRATCH; reference, do NOT commit) ----
  exp_log/
    logs/*.out, *.err    # job stdout/stderr
    checkpoints/*.pt     # model checkpoints (often GBs)
    tensorboard/         # TB event files
    wandb/               # W&B local run dir
    wandb_run_id.txt     # W&B run id (link back to the dashboard)
    .done.json           # optional self-report marker (harvest fallback: {status, finished_at, best_metric})
  _smoke/                # smoke-test outputs (same shape as exp_log/; throwaway)
docs/
  experiments.md             # DERIVED index — regenerated by /harvest from all metadata.yaml; never hand-edited
  experiments/<run_code>.md  # human NARRATIVE (Goal/Setup/Results/Observations) — md = story, yaml = metadata
```

## `metadata.yaml` schema

### Core (always write)

| Field | Meaning |
|---|---|
| `run_code` | unique, filesystem-safe id (also the folder name) |
| `status` | `submitted｜running｜completed｜failed｜timeout｜cancelled｜oom` |
| `tags` | list of keywords |
| `cluster`, `account` | where it ran / under which allocation |
| `git: {commit, dirty, diff_path}` | provenance; `diff_path` → saved diff file when `dirty: true` |
| `slurm_job_ids` | list (smoke + train + eval ids) |
| `started_at`, `finished_at` | ISO-8601 |
| `objective: {goal, expected_result: {metric, value, rationale}}` | what success looks like, pre-registered |
| `decision_rule` | what each outcome (above/at/below expected) will mean |
| `best_metric: {name, value, epoch}` | the headline result |
| `notes` | free text for a human reader |

### Reproducibility (strongly recommended when a run must be re-creatable)

| Field | Meaning |
|---|---|
| `config.yaml` + `config_resolved.yaml` | snapshot the input AND the resolved config (files, not metadata fields) |
| `base_checkpoint: {path, sha256}` | starting weights + hash |
| `data_split_version` | dataset/split tag or hash |
| `env_hash` | hash of the environment (pair with a `pip freeze` / `requirements.txt` snapshot) |
| `wandb_run_id` | W&B run id (or store `exp_log/wandb_run_id.txt`) |

### Scientific / lineage (optional — adopt as the work warrants; per-person)

| Field | Meaning |
|---|---|
| `stage`, `base` | grouping / base config name |
| `parent_run`, `superseded_by` | lineage back- and forward-pointers |
| `hypothesis` | the testable claim this run probes |
| `assumptions: [{statement, validation, risk_if_false}]` | what must hold for the result to be meaningful |
| `objective.expected_intermediate_signals: [{at_epoch, signal, expected_range, action_if_missed}]` | early-abort signals |

## Complete example `metadata.yaml`

A full, well-formed record (illustrative values; structure mirrors a real training run):

```yaml
# ---- Core ----
run_code: A2_03_sft_lora_r16
status: completed
tags: [sft, lora-r16, llama3-8b, bs8, 3ep]
cluster: fir
account: rrg-<pi>_gpu
git:
  commit: 3bd7d07a779715ac8d53560a4e612078dbddb62e
  dirty: true
  diff_path: git.diff            # saved working-tree diff (dirty run still reproducible)
slurm_job_ids: ['41061405']
started_at: '2026-05-22T20:59:33Z'
finished_at: '2026-05-22T21:48:42Z'
objective:
  goal: Does LoRA rank 16 recover full-fine-tune quality at a fraction of the cost?
  expected_result: { metric: eval_loss, value: 1.20, rationale: "FT baseline = 1.18; LoRA-r8 = 1.31" }
decision_rule: ">=1.30 LoRA underfits (raise rank); 1.20-1.30 acceptable; <=1.20 matches FT (ship r16)"
best_metric: { name: eval_loss, value: 1.214, epoch: 3 }
notes: "Fork of r8 run; only lora_rank 8->16. bf16, grad_clip 1.0."
# ---- Reproducibility ----
base_checkpoint: { path: checkpoints/llama3-8b-base.pt, sha256: c1a54157f93df34f3eb702d19e0a03b2 }
data_split_version: alpaca_clean_v2
env_hash: bc39458c7ef054f7c0579e6b67fabf79
wandb_run_id: 5ph9v1yv
# ---- Scientific / lineage (optional) ----
stage: A2
base: sft_lora_base
parent_run: A2_02_sft_lora_r8
superseded_by: null
hypothesis: Rank 16 closes most of the r8->full gap because the adapted layers are rank-bound, not depth-bound.
assumptions:
  - statement: The eval set is disjoint from the SFT data.
    validation: hash overlap check = 0.
    risk_if_false: eval_loss is optimistic; result meaningless.
objective.expected_intermediate_signals:
  - { at_epoch: 1, signal: train_loss, expected_range: "< 1.6", action_if_missed: "abort — adapter not learning" }
```

## Artifact table — keep vs reference

| Artifact | Where | In the durable record? |
|---|---|---|
| `metadata.yaml`, `config.yaml`, `config_resolved.yaml`, `slurm_*.sh`, `git.diff` | run folder | ✅ keep (small) |
| `eval_<variant>/*.csv` | run folder | ✅ keep (small) |
| checkpoints, tensorboard, wandb blobs | `exp_log/` in `$SCRATCH` | ❌ reference only (path + sha256); regenerable / GBs |
| `wandb_run_id.txt` | `exp_log/` | ✅ keep the id (1 line) |
| `_smoke/` | run folder | ❌ throwaway |

## SSOT rules

- `metadata.yaml` is the only place status lives. `docs/experiments.md` is a **derived** view —
  `harvest` regenerates it; never parse it for truth, never hand-edit it.
- Record real provenance: when the tree is dirty, set `dirty: true` and save the diff to
  `git.diff` (`git.diff_path`). Never claim a clean commit you cannot reproduce.
- Never fabricate a metric or a terminal status — only write one you actually resolved.
